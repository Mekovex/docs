Дымовой набор тестов перед выпуском релиза

Цель
Данный дымовой набор подтверждает, что продукт **«Сервис задач»** находится в работоспособном состоянии и может быть передан на дальнейшее тестирование или опубликован. Набор проверяет **критический путь** и два типовых риска сопровождения, но не заменяет полное регрессионное тестирование.

---

Подготовка окружения
**Перед выполнением проверок обязательно выполните:**
- Версия Python: **3.8–3.11**
- Активировать виртуальное окружение:
  ```bash
  source .venv/bin/activate   # Linux/macOS
  .venv\Scripts\activate      # Windows
  ```
- Установить зависимости:
  ```bash
  pip install -r requirements.txt
  ```
- Команда запуска проекта:
  ```bash
  python src/app.py
  ```
- Путь к файлу данных (если используется): `data/tasks.json` (создаётся автоматически)

**Если окружение настроено иначе — проверки невалидны!**

---

3. Критерии решения
- **Релиз РАЗРЕШЁН**, если все проверки уровня **SMK-01 … SMK-07** с пометкой **BLOCK** успешно пройдены.
- **Релиз ЗАПРЕЩЁН**, если **не проходит хотя бы одна проверка с пометкой BLOCK**.

---

4. Минимальный набор проверок

SMK-01: Проверка окружения и зависимостей
- **Тип:** окружение  
- **Важность:** BLOCK  
- **Шаги:**
  1. Активировать виртуальное окружение.
  2. Выполнить `python --version`.
  3. Выполнить `pip install -r requirements.txt`.
- **Ожидаемый результат:**
  - Версия Python соответствует требованиям проекта.
  - Зависимости устанавливаются без ошибок (конфликтов версий).

---

SMK-02: Запуск приложения
- **Тип:** запуск  
- **Важность:** BLOCK  
- **Шаги:**
  1. Запустить `python src/app.py`.
- **Ожидаемый результат:**
  - Приложение стартует без исключений.
  - Отображается стартовое меню с доступными командами.

---

SMK-03: Создание задачи (критический путь)
- **Тип:** функциональная  
- **Важность:** BLOCK  
- **Шаги:**
  1. В главном меню выбрать команду создания задачи.
  2. Ввести название: `Smoke task`.
  3. Ввести описание: `Release testing`.
- **Ожидаемый результат:**
  - Задача создана, получен её идентификатор (например, `ID: 1`) или подтверждение `"Задача добавлена"`.

---

SMK-04: Просмотр списка задач (критический путь)
- **Тип:** функциональная  
- **Важность:** BLOCK  
- **Шаги:**
  1. В главном меню выбрать команду вывода списка задач.
- **Ожидаемый результат:**
  - В списке присутствует задача с названием `Smoke task`.

---

SMK-05: Изменение статуса задачи (критический путь)
- **Тип:** функциональная  
- **Важность:** BLOCK  
- **Шаги:**
  1. Выбрать команду изменения статуса.
  2. Указать идентификатор созданной задачи.
  3. Установить статус `done`.
- **Ожидаемый результат:**
  - Статус задачи изменён.
  - При повторном просмотре списка статус отображается как `done`.

---

SMK-06: Сохранение данных (критический путь)
- **Тип:** функциональная  
- **Важность:** BLOCK  
- **Шаги:**
  1. Выполнить команду сохранения данных (если требуется явно, или проверить автоматическое сохранение при выходе).
  2. Проверить, что файл `data/tasks.json` существует и не пуст.
- **Ожидаемый результат:**
  - Файл данных создан/обновлён.
  - В файле присутствует запись о задаче `Smoke task`.

---

SMK-07: Перезапуск и восстановление данных (критический путь)
- **Тип:** функциональная  
- **Важность:** BLOCK  
- **Шаги:**
  1. Закрыть приложение (Ctrl+C или выход из меню).
  2. Запустить приложение снова (`python src/app.py`).
  3. Вывести список задач.
- **Ожидаемый результат:**
  - Задача `Smoke task` сохранена, присутствует в списке со статусом `done`.

---

SMK-08: Поиск задачи (дополнительная функция)
- **Тип:** функциональная  
- **Важность:** WARN  
- **Шаги:**
  1. Выбрать команду поиска задач.
  2. Ввести ключевое слово `Smoke`.
- **Ожидаемый результат:**
  - Поиск возвращает задачу `Smoke task`.

---

SMK-09: Обработка ошибочного ввода (риск №1)
- **Тип:** ошибка  
- **Важность:** WARN  
- **Шаги:**
  1. Выбрать команду создания задачи.
  2. Ввести **пустое название** (нажать Enter без текста).
- **Ожидаемый результат:**
  - Приложение выдаёт понятное сообщение об ошибке (`"Название не может быть пустым"`).
  - Данные не повреждены, существующие задачи сохранились.

---

SMK-10: Работа при отсутствии файла данных (риск №2)
- **Тип:** ошибка  
- **Важность:** WARN  
- **Шаги:**
  1. Выйти из приложения.
  2. Переименовать файл `data/tasks.json` в `data/tasks_backup.json`.
  3. Запустить приложение заново.
  4. Вывести список задач.
- **Ожидаемый результат:**
  - Приложение не падает с ошибкой.
  - Создаётся новый файл `data/tasks.json` (или отображается сообщение, что список задач пуст).

---

5. Приоритет и порядок выполнения
Проверки выполняются **строго в указанном порядке** — от самых дешёвых к самым дорогим:

1. **SMK-01** (окружение)
2. **SMK-02** (запуск)
3. **SMK-03 … SMK-07** (критический путь)
4. **SMK-08** (дополнительно)
5. **SMK-09 … SMK-10** (ошибочные ситуации)

---

6. Пояснение выбора проверок BLOCK

SMK-01 (Проверка окружения) — BLOCK
**Почему запрещает релиз:**  
Если окружение не настроено или зависимости не установлены, приложение **невозможно запустить**. Релиз теряет смысл, так как пользователь не сможет даже начать работу.

SMK-07 (Перезапуск и восстановление данных) — BLOCK
**Почему запрещает релиз:**  
Если данные не сохраняются или теряются при перезапуске, это приводит к **полной потере пользовательских данных**. Такой релиз опасен, так как пользователь ожидает, что задачи будут доступны после перезапуска программы.
